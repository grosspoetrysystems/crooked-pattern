# Plan Execution Lifecycle

This skill defines the ceremonies for executing Tempor-generated plans through phased delivery with human-in-the-loop orchestration.

Entry-point note:
- when execution starts immediately after `tempor.decompose_plan`, use `post-council-handoff` first to establish the execution anchor
- once the active phase, first wave, and initial execution contract are clear, this skill governs the ongoing lifecycle

The lifecycle borrows from agile ceremony patterns — planning, grooming, review, retrospective — but adapted to plan-driven delivery where the backlog is a deliberated plan, waves are sprints, and gates are reviews.

**Important boundaries:**
- **Generator vs executor.** This is an execution-side concern. Tempor generates plans via multi-model deliberation. This skill governs how those plans are consumed, tracked, and evolved through delivery phases. Do not conflate this with Tempor's generator-only responsibilities.
- **Anchor vs lifecycle.** `post-council-handoff` is the explicit bridge from decomposition into execution. This skill takes over once the initial execution frame is established.
- **Repo-local, not application source.** This skill lives in `.agents/skills/` — it governs how agents and humans work in this repository. It is not part of the Tempor application (`src/`) and is not shipped to users.
- **Product language potential.** These ceremony patterns (kickoff, gate, closeout, lookahead rotation) could inform Tempor's product surface — how generated plans are structured for consumption — without crossing the generator-only boundary.

## Note on Plan Lineage

In this repository, v1-v4 were generated by Tempor through multi-model deliberation. v5+ forked from those into execution-generated documents that follow the lookahead/kickoff pattern described here. They share a version namespace but represent a separate lineage — execution scaffolding derived from, but not produced by, the deliberation process.

## Ceremony Cycle

```
lookahead ──→ kickoff ──→ waves ──→ closeout ──→ next lookahead becomes kickoff ──→ ...
```

Each phase follows this ceremony rotation:

1. **Grooming** — a lookahead exists from the previous phase (or is created at project start). It is the unrefined backlog for the upcoming phase.
2. **Kickoff** — the lookahead is rebaselined into a kickoff, adjusted with learnings, statuses, and new context from the closeout. This is the planning ceremony that commits the phase scope.
3. **Execution** — the kickoff is delivered in waves. Each wave is a sprint with parallel work items.
4. **Gate review** — at the end of each wave, a HITL gate acts as a sprint review. Work is inspected, deferred items are documented, and the next wave is approved.
5. **Closeout** — the retrospective. Captures what was done, what was deferred, what was learned. Archives the phase.
6. **Rotation** — the existing lookahead for the next phase is groomed into the new kickoff. A new lookahead is created for the horizon beyond that.

## Mandatory Cadence

Operating imperative:

- Lifecycle execution is artifact-first at both the phase and wave levels.
- Do not treat a phase kickoff/lookahead pair as sufficient once a wave is active.
- Before executing or closing a wave, confirm the active wave kickoff and next-wave lookahead exist and are linked from the plan index.
- If either wave artifact is missing, create or repair it before continuing implementation.

Ceremonies run at both levels:

1. **Wave-end gate ceremony (required):** tactical coordination for blockers, sequencing, and local drift correction.
2. **Phase-end closeout ceremony (required):** structural review, carry-forward decisions, and transition rebaseline.

Phase artifacts are required, not optional:

1. Every phase must have a kickoff document.
2. Every phase must maintain a lookahead for the next horizon.
3. At transition, prior lookahead is rebaselined into kickoff and a new lookahead is created/updated.

Wave artifacts are also required:

1. Every active wave must have a wave kickoff document.
2. Every active wave must maintain a next-wave lookahead.
3. At wave gate, the next-wave lookahead is either rebaselined into the next wave kickoff or retired when the phase closes.

Skill attachment requirement:

1. Every wave gate and phase closeout ceremony must explicitly attach `git-commit-workflow`.
2. The ceremony artifact should include a short `Skill attachments` section listing:
- `git-commit-workflow` (required)
- additional skills used for that ceremony (optional)

Product impact briefing requirement:

1. Every wave gate and phase closeout must include a short product-impact briefing tied to the active plan.
2. The briefing must state:
- active `Plan/Phase/Wave/Track` context and objective
- key product benefits unlocked by delivered scope
- key tradeoffs/costs/risks accepted (and why)
- explicit linkage to planning artifacts used for the decision (`PLAN_INDEX*`, active kickoff, lookahead, canonical decomposition/source docs)
3. Keep this briefing implementation-focused and concise; avoid generic process-only language.

## Parallel Orchestration Doctrine (Worktree-First)

Full doctrine: [references/orchestration-protocol.md](references/orchestration-protocol.md)

Planning and decomposition artifacts must treat Git worktrees as a first-class execution primitive for parallel subagent delivery. Key requirements:

1. Per-track workspace isolation strategy (worktree path/branch naming)
2. Integration boundary per track (merge surface, shared contracts, conflict hotspots)
3. Deterministic evidence chain per track (commit refs, changelog, gate evidence, drift signals)
4. Orchestrator handoff contract per wave (inputs, outputs, promotion criteria, rollback posture)

The doctrine defines execution modes (background-parallel, foreground-sequential, orchestrator-direct), checkpoint tiers (post-task, mid-task, per-call, continuous), sensitive area flags, and a pre-launch checklist for each wave.

## 1) Lookahead (Grooming)

A lookahead is the unrefined backlog for a future phase. It is written with incomplete information and is expected to change — that's the point. It gives shape to the next horizon so the kickoff ceremony has material to work from.

**Create a lookahead when:**
- Closing the current phase (always create the next-next horizon)
- Starting a project (create both the first kickoff and the first lookahead)

**A lookahead contains:**
- Scope: task areas from the master plan (v2/v4 or equivalent)
- Dependency order: what must land before what
- Proposed waves: parallel work groupings with HITL gates
- Entry criteria: what must be true before this phase can start
- Exit criteria: what must be true to close this phase
- Key unknowns: what we don't know yet that execution will reveal

**A lookahead does NOT contain:**
- Rebaseline adjustments (that happens at kickoff)
- Detailed task status (not started yet)
- Reference commits (no code has landed)

**Naming:** `v{N}-phase{P}-lookahead.md`

## 2) Kickoff (Planning Ceremony)

A kickoff is the planning ceremony where a lookahead is rebaselined with concrete context. It commits the phase scope and becomes the governing document for execution.

**Create a kickoff when:**
- The previous phase is closed and the next phase is ready to start
- The lookahead for this phase already exists and needs rebaseline

**Rebaseline inputs:**
- The existing lookahead for this phase
- Closeout status from the previous phase (what completed, what deferred, what learned)
- Current codebase state (reference commit)
- All generated docs from previous work (architecture decisions, content notes, changelogs)
- Deferred tasks carried forward from previous phases
- Updated dependency analysis based on what actually exists now vs what was assumed

**A kickoff contains:**
- Everything from the lookahead, plus:
- Reference commit
- Task status matrix (what's done, partial, not started)
- Rebaseline rationale (why this differs from the lookahead)
- Carried-forward items from previous phases with provenance
- Updated waves adjusted for current reality

**Naming:** `v{N}-phase{P}-kickoff.md` (or `v{N}-phase{P}-rebaseline-kickoff.md` if a mid-phase rebaseline)

## 3) Waves (Sprints) and Gates (Reviews)

Waves are sprints — the unit of parallel work within a phase. Each wave ends with a HITL gate that serves as the sprint review.

**Wave rules:**
- Parallel work items within a wave have no dependencies on each other
- Sequential dependencies go into separate waves
- Each wave has named tracks (T-* prefix, e.g., T-DB-BOUNDARY, T-AUTH-FLOW). Use descriptive names, not opaque identifiers.
- One agent owns one track. Do not create dedicated integrator or QA tracks — those are orchestrator responsibilities.
- Each active wave has its own kickoff document and next-wave lookahead.
- Each wave has a gate checklist
- Quality gates (`pnpm check`, `pnpm test`, `pnpm test:coverage:check`) are part of every gate

### Delivery order within a wave

Work items follow a docs-first, tests-first discipline:

1. **Docs first.** Write or update documentation before writing implementation code. This validates the design with stakeholders, catches scope issues early, and prevents rework. Architecture decisions, API contracts, and user-facing docs land before the code that implements them.

2. **Tests first.** Write or update tests before or alongside implementation. Tests define the acceptance criteria in code. New behavior should have test coverage before the implementation is considered complete.

3. **Implementation.** Write the code that satisfies the docs and tests.

4. **Commit via `git-commit-workflow` skill.** Every meaningful deliverable is committed using the repo's commit workflow skill. Commit history is the execution record — it captures what was done, in what order, with changelog and content note traceability.

### Commit history as execution record

The git commit history serves as the ground truth for wave execution. Each commit captures a discrete deliverable with:
- Conventional message (`feat:`, `fix:`, `chore:`)
- Changelog entry with changeset ID
- Content note cross-reference for non-trivial work

At HITL gates, review the commit log since the last gate to verify all wave items were delivered and documented. Use `git log --oneline` as the execution audit trail.

Commit focus rule:
- commits should primarily document `src/` product changes and their direct tests/docs;
- closeout or other ceremony checkpoints are sensible commit timing markers, but they do not require standalone closeout-document commits unless explicitly requested.

### HITL gates check:
- Docs for wave items written or updated
- Tests for wave items written and passing
- Work items meet acceptance criteria
- No regressions introduced
- Deferred items explicitly documented with rationale
- Commit history is clean and traceable (follows `git-commit-workflow`)
- Quality gates green
- Product-impact briefing included (benefits/tradeoffs tied to plan artifacts)
- `.tempor/PULSE_CHECK.md` refreshed via `pulse-save` after gate evidence, quality gates, and commit
- Human approves before next wave starts

**Mid-phase rebaseline:** If a wave reveals significant scope change, create a rebaseline kickoff (new version) before continuing. Do not silently drift from the governing kickoff.

## Drift Evaluation and Re-Council Trigger

Ceremonies are not only progress checkpoints; they are drift checkpoints.

At every HITL gate, evaluate:

1. **Scope drift:** Does delivered work still align with the active kickoff intent?
2. **Decision drift:** Are decisions/assumptions changing repeatedly without convergence?
3. **Context-window pressure:** Are teams losing track of critical context (unknowns resurfacing, repeated re-explaining, incoherent carry-forward)?

Escalation policy:

1. At wave level, apply tactical fixes first (blocker sequencing, bounded scope correction, wave reshaping).
2. Trigger a fresh Tempor council deliberation only when drift is sustained across phase execution or when the current planning document is no longer coherent/fit for purpose.
3. Rebaseline the active phase kickoff from council output when re-council is triggered.
4. If context-window pressure or local drift makes the current session unreliable before a formal gate, run `pulse-save` as a manual eject checkpoint before continuing in a fresh context window.
4. Document trigger rationale and resulting changes in closeout/kickoff artifacts.

Important boundary:

- This is local execution best practice for this repository.
- It is not a Tempor `src/` product feature requirement.

## 4) Closeout (Retrospective)

The closeout is the phase retrospective — it captures what was delivered, what was deferred, and what was learned before transitioning to the next phase.

**Closeout actions:**
1. Update the kickoff doc with final task statuses
2. Document what was completed, what was deferred (with rationale), and what was learned
3. Archive the closed phase's documents into `archive/`
4. Update `PLAN_INDEX.md` — move phase to Past with summary, update Present pointers
5. Create a signed closeout commit with all deliverables
6. Verify the next phase's lookahead still makes sense given learnings
7. Create a new lookahead for the horizon beyond the next phase

**Closeout feeds into the next kickoff:**
- Task completion statuses
- Deferred task specs with full context for carry-forward
- Architecture decisions made during execution
- Updated assumptions and resolved unknowns
- Generated docs (changelogs, content notes, new product docs)
- Product benefit/tradeoff summary linked to active planning artifacts

## 5) Rotation (Backlog Refinement)

At every closeout, two backlog refinement actions happen:

1. **The existing next-phase lookahead** is rebaselined into the new kickoff (see step 2).
2. **A new lookahead** is created for the phase after that — maintaining a one-phase-ahead planning horizon.

This ensures:
- You always have visibility into the next horizon
- Lookaheads improve over time as each phase's learnings feed forward
- No phase starts cold — there's always a draft to rebaseline from

## Plan Index as Source of Truth

The plan directory maintains a `PLAN_INDEX_V{N}.md` that serves as the source of truth for the lifecycle, where `{N}` is the version of the meta plan generated by the council (e.g., v2 = the deliberated task definitions). It organizes all plan documents into three sections:

- **Present:** the active kickoff and active lookahead — these are the live, evolving documents
- **Present wave:** the active wave kickoff and active wave lookahead — these are required whenever a wave is active
- **Past:** closed phases with summary, closeout commit reference, and pointer to archived docs
- **Future:** upcoming phases with pointers to their kickoff/lookahead docs

### Index invariants (must hold)

1. **Single active phase:** `Present` points to exactly one active phase kickoff and one active phase lookahead.
2. **Single active wave:** `Present` points to exactly one active wave kickoff and one active wave lookahead.
3. **No cross-section duplication:** a phase can appear in only one section at a time.
4. **Future starts at next phase:** once a phase is active in `Present`, it must not remain in `Future` as `NEXT`.
5. **Past is closed-only:** only closed phases belong in `Past`.

Reusable starter material:

- `.agents/skills/plan-execution-lifecycle/references/plan_index_template.md`

Recommended index annotation style:

- Keep canonical execution IDs (`P*-W*`, `T-*`) unchanged.
- If needed, add brief explanatory language using the crosswalk reference without renaming artifacts.

### File layout

```
{plan_id}/
  PLAN_INDEX_V{N}.md         ← source of truth for meta plan version N
  {active-phase-kickoff}.md  ← current phase, evolving
  {active-phase-lookahead}.md← next phase horizon, evolving
  {active-wave-kickoff}.md   ← current wave, evolving
  {active-wave-lookahead}.md ← next wave horizon, evolving
  archive/                   ← closed phase documents
    {closed-kickoff}.md
    {closed-lookahead}.md
    {closed-wave-kickoff}.md
    {closed-wave-lookahead}.md
    {closed-rebaseline}.md
    PLAN_INDEX_V{N-1}.md     ← archived index from previous meta plan
    ...
```

### Archive rules

At phase closeout, move all documents belonging to the closed phase into `archive/`. The index retains a summary of each closed phase (tasks, closeout commit, key outcomes) so the archive is navigable without reading every file. Only the active kickoff and active lookahead remain at the top level.

At meta plan closeout (all phases of a deliberated plan complete), archive the entire `PLAN_INDEX_V{N}.md` along with its remaining top-level documents. When a new meta plan is deliberated (e.g., v3 via a fresh Tempor council run), create `PLAN_INDEX_V{N+1}.md` as the new source of truth. The previous index and its execution history are preserved in `archive/` for reference.

Source plans (Tempor-generated deliberation outputs) live at the top level as permanent reference — they are not archived.

### Document lifecycle example

```
v4  — phased decomposition (master plan, top-level permanent)
v11 — phase 3 lookahead → archived at phase 3 closeout
v12 — phase 4 lookahead → rebaselined into v14 kickoff, then archived
v13 — phase 3 kickoff/rebaseline → archived at phase 3 closeout
v14 — phase 4 kickoff (active)
v15 — phase 5 lookahead (active)
v16 — phase 5 kickoff (future, created at phase 4 closeout)
v17 — phase 6 lookahead (future, created at phase 4 closeout)
```

## Generator vs Executor Boundary

This skill is executor-side orchestration. The boundary:

| Concern | Owner |
|---------|-------|
| Plan generation (contributor proposals, prime synthesis, consensus) | Tempor (generator) |
| Plan iteration and versioning | Tempor (generator) |
| Cost estimation and model selection | Tempor (generator) + `docs/COST_ANALYSIS.md` |
| Phase scoping, wave design, HITL gates | This skill (executor) |
| Task execution (code, tests, docs) | Agents + humans (executor) |
| Closeout evaluation and rebaseline decisions | Humans (executor, HITL) |
| Lookahead creation for next horizon | This skill (executor) |

Tempor generates plans. This skill governs how those plans move through delivery. Do not add execution concerns to Tempor. Do not add generation concerns to this skill.

## Checklist: Phase Transition Ceremony

Use this when performing a closeout + kickoff transition:

- [ ] All wave gates passed or items explicitly deferred
- [ ] Deferred items documented with rationale and carry-forward specs
- [ ] Architecture decisions captured in relevant docs
- [ ] Changelog and content notes updated
- [ ] Quality gates green
- [ ] Closed phase documents moved to `archive/`
- [ ] `PLAN_INDEX.md` updated (phase moved to Past, Present pointers updated)
- [ ] `PLAN_INDEX.md` invariants hold:
  - [ ] active phase appears only in `Present`
  - [ ] active phase kickoff and active phase lookahead are listed
  - [ ] active wave kickoff and active wave lookahead are listed when a wave is active
  - [ ] no phase appears in multiple sections
  - [ ] `Future` starts at next not-yet-active phase
- [ ] Closeout commit signed with all deliverables
- [ ] Existing lookahead for next phase rebaselined into kickoff
- [ ] New lookahead created for the phase after next
- [ ] Key unknowns updated (resolved ones removed, new ones added)
- [ ] Entry criteria for next phase verified
- [ ] Ceremony terms in index + active kickoff/lookahead docs are consistent with crosswalk disambiguation
